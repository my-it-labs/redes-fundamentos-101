# Laboratorio M03-01 — DNS

[← Página anterior](../M02/M02-03-ipv6-basico.md) · [Siguiente página →](M03-02-dhcp.md)

## Objetivo del laboratorio

Al terminar debes poder:

- Consultar un servidor DNS con `dig` indicando el resolver (`@IP`).
- Interpretar registros **A** y **CNAME** en una zona de laboratorio.
- Relacionar nombre (`intranet.lab`) con IP del servicio web de la maqueta.

En cada paso: **levantar la maqueta** → **acceder al sistema** → comandos **dentro del sistema**.

Conceptos: [Glosario de términos](../../docs/glosario-terminos.md) (DNS) · Comandos: [Glosario de herramientas](../../docs/glosario-herramientas.md).

---

## Mapa mental (antes de tocar comandos)

```text
cliente-dns ──consulta──► servidor-dns (192.168.52.2 / dnsmasq)
                              │
                              └──► intranet.lab → 192.168.52.20 (servidor-web)
```

| Nombre | Tipo en dnsmasq | Resultado esperado |
|--------|-----------------|-------------------|
| `intranet.lab` | A | `192.168.52.20` |
| `www.intranet.lab` | alias a la misma IP | `192.168.52.20` |

---

### Paso 1 — Levantar la maqueta DNS

**Aprende:** un resolver autoritativo **local** (dnsmasq) responde solo los nombres definidos en su configuración.

**Levantar la maqueta:**

```bash
cd labs/M03/compose/dns
docker compose up -d
```

**Acceder al sistema `cliente-dns`:**

```bash
docker compose exec -it cliente-dns bash
```

**Dentro del sistema `cliente-dns`:**

```bash
dig @192.168.52.2 intranet.lab +short
```

**Deberías ver:** `192.168.52.20`

**Por qué:** `dnsmasq.conf` mapea `intranet.lab` a la IP de `servidor-web`.

---

### Paso 2 — Consulta detallada y CNAME

**Aprende:** `dig` sin `+short` muestra sección ANSWER, TTL y tipo de registro.

**Dentro del sistema `cliente-dns`:**

```bash
dig @192.168.52.2 intranet.lab
dig @192.168.52.2 www.intranet.lab +short
```

**Deberías ver:** en la primera consulta, registro **A**; en la segunda, la misma IP (CNAME resuelto a `web.intranet.lab` → A).

**Dentro del sistema `cliente-dns`:**

```bash
dig @192.168.52.2 alias.intranet.lab +short
```

**Deberías ver:** `192.168.52.30` (definido en DNS; puede no haber host en la maqueta — el nombre **resuelve** igual).

---

### Paso 3 — Comprobar conectividad al nombre

**Aprende:** resolver un nombre no implica que el servicio responda; hace falta IP correcta **y** host activo.

**Dentro del sistema `cliente-dns`:**

```bash
ping -c 2 192.168.52.20
ping -c 2 intranet.lab
```

Si `ping intranet.lab` falla por falta de resolver en el cliente, usa:

```bash
getent hosts intranet.lab || true
ping -c 2 $(dig @192.168.52.2 intranet.lab +short)
```

**Deberías ver:** ping OK hacia `.20` (servidor-web levantado).

**Dentro del sistema:** `exit`

**En tu terminal (maqueta):** `docker compose down`

---

### Paso 4 — Reverso conceptual (opcional)

**Aprende:** **PTR** asocia IP → nombre; en laboratorio pequeño a menudo no está configurado.

**Acceder al sistema `cliente-dns`:** `docker compose up -d` y `exec` como antes.

**Dentro del sistema `cliente-dns`:**

```bash
dig @192.168.52.2 -x 192.168.52.20 +short
```

**Deberías ver:** vacío o sin PTR (normal en esta maqueta).

**Dentro del sistema:** `exit`

**En tu terminal (maqueta):** `docker compose down`

---

## Antes de seguir

### Pon el foco en

| Elemento | Rol |
|----------|-----|
| Resolver (`@192.168.52.2`) | A quién preguntas |
| Registro A | Nombre → IPv4 |
| CNAME | Alias → otro nombre |
| TTL | Cuánto se cachea la respuesta |

### Reto

**1. Nombre inexistente** — Consulta `noexiste.intranet.lab` con `dig @192.168.52.2`.

<details>
<summary>Ver solución</summary>

**Dentro de `cliente-dns`:**

```bash
dig @192.168.52.2 noexiste.intranet.lab
```

**Deberías ver:** `status: NXDOMAIN` o sin sección ANSWER.

</details>

**2. Registro nuevo** — Añade en `dnsmasq.conf` `address=/api.intranet.lab/192.168.52.20`, reinicia el compose y consulta con `dig`.

<details>
<summary>Ver solución</summary>

Tras editar el fichero:

**En tu terminal (maqueta):** `docker compose up -d --force-recreate servidor-dns`

**Dentro de `cliente-dns`:** `dig @192.168.52.2 api.intranet.lab +short` → `192.168.52.20`

</details>

**3. Comparar con resolver del host** — En tu **terminal del Codespace**, `dig intranet.lab +short` (sin `@`). ¿Coincide con el lab?

<details>
<summary>Ver solución</summary>

Suele **no** coincidir: el Codespace usa DNS del proveedor, no `192.168.52.2`. Por eso en el lab siempre indicas `@192.168.52.2`.

</details>
